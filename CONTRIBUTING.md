# Contributing to MatrixNet

Thanks for your interest in MatrixNet. This is an early-stage, **Phase 1** project
under active development — the architecture is settled but the surface area is
still moving, so coordination matters. Please read this guide before opening an
issue or a pull request.

By participating in this project you agree to abide by our
[Code of Conduct](./CODE_OF_CONDUCT.md).

## Before you start

- For anything non-trivial, **open an issue first** to discuss the approach. This
  avoids duplicate work and wasted effort while internals are still in flux.
- Read [`docs/ARCHITECTURE.md`](./docs/ARCHITECTURE.md) to understand the
  passive, dual-source design and the module boundaries.
- Scope your change. The pure-logic core, the privileged helper, and the UI have
  very different review requirements — keep a PR to one concern where possible.

## Development environment

| Tool | Version / notes |
|---|---|
| macOS | 26 (Tahoe) or later |
| Xcode | 26 |
| [XcodeGen](https://github.com/yonaskolb/XcodeGen) | generates the Xcode project from `project.yml` |
| [SwiftLint](https://github.com/realm/SwiftLint) | lint (configured by `.swiftlint.yml`) |
| [SwiftFormat](https://github.com/nicklockwood/SwiftFormat) | formatting (configured by `.swiftformat`) |

The pure-logic core lives in a local Swift Package and is fully testable with
`swift test` — no Xcode required. The macOS app and the privileged capture helper
are Xcode targets generated by XcodeGen.

```sh
# Generate the Xcode project (App + helper targets)
xcodegen generate
```

## Building and running tests

The core libraries are designed so that all logic can be exercised without the
app or the helper:

```sh
# Run the full core test suite
swift test

# Run the concurrency-sensitive suites under ThreadSanitizer
swift test --sanitize=thread

# Run the dissection / parsing suites under AddressSanitizer
# (recommended when touching parsers or feeding malformed input)
swift test --sanitize=address
```

> Exact targets, filters, and any helper/UI integration test commands are **to be
> finalized** as those layers land. Until then, prefer adding logic to the core
> package where it can be tested directly with `swift test`.

Before pushing, make sure lint and formatting pass:

```sh
swiftformat --lint .
swiftlint
```

## Test-driven development (required)

MatrixNet is built **test-first**. We use Apple's
[Swift Testing](https://developer.apple.com/documentation/testing) framework
(`import Testing`; `@Test`, `#expect`, `#require`, parameterized
`@Test(arguments:)`, `confirmation` for async event counts, `@Suite`, and
`.timeLimit`). Fall back to XCTest only where Swift Testing does not yet cover a
scenario (e.g. certain UI or performance-baseline tests).

The expected workflow for a feature or bugfix:

1. Write a failing test that captures the desired behavior (or reproduces the
   bug).
2. Implement the minimum to make it pass.
3. Refactor with the tests green.

PRs that add or change behavior without tests will be asked to add them.

### Test coverage expectations

Two areas get special scrutiny, because they are where this kind of tool breaks:

- **Boundary / malformed input (parsers).** Every dissector must be hostile-input
  safe. Systematically cover empty input, zero-length fields, single bytes,
  truncation at each header boundary, length fields that exceed the buffer
  (overflow / out-of-bounds), zero or maximal length fields, illegal
  version/protocol numbers, self-referential or out-of-bounds TLV options,
  fragmentation/reassembly edges, TCP sequence wraparound, duplicates and
  reordering, and unaligned offsets. The rule: **no malformed input may crash,
  loop forever, or read out of bounds** — return a partial parse or an explicit
  error. Fuzz-style mutated input is encouraged.
- **Concurrency / race conditions.** The capture-ingest actor and the correlation
  engine must stay correct under concurrent, high-rate ingestion from both
  sources at once (NetworkStatistics events and PKTAP packets). Cover lifecycle
  races (added/counts/removed interleaved with packet arrival, PID reuse, process
  exit), and assert no data races, no loss, and no duplication. Use
  `confirmation(expectedCount:)` to assert exact async callback counts and
  `.timeLimit` to guard against deadlocks. CI runs the concurrency suites under
  ThreadSanitizer.

## Code style

- All public code, comments, and DocC documentation are written in **English**.
- Swift 6 language mode with **strict concurrency**. Respect actor isolation;
  the UI runs on the main actor, capture/parsing on background actors.
- **SwiftLint and SwiftFormat must pass.** Do not disable rules to work around a
  warning without justification in the PR.
- Keep modules dependency-correct: the dependency graph in
  [`docs/ARCHITECTURE.md`](./docs/ARCHITECTURE.md) must stay acyclic. Logic that
  can live in a pure module should live there, not in the app or the helper.

## Commit and branch conventions

- Work on a feature branch; do not commit directly to `main`.
- Commit messages follow [Conventional Commits](https://www.conventionalcommits.org/):

  ```
  <type>(<optional scope>): <description>

  [optional body]

  [optional footer(s)]
  ```

  Common types: `feat`, `fix`, `docs`, `test`, `refactor`, `perf`, `build`,
  `ci`, `chore`. Examples:

  ```
  feat(dissection): add TLS ClientHello SNI extraction
  fix(correlator): handle PID reuse on rapid connection churn
  test(pcap): cover truncated pcapng section header round-trip
  ```

- Keep commits focused and the history readable; squash noise before review.

## Pull request process

1. Ensure `swift test`, `swiftlint`, and `swiftformat --lint .` all pass locally.
2. Open the PR against `main` with a clear description of the change, the
   motivation, and how it was tested. Link the related issue.
3. Update documentation (`README.md`, `docs/ARCHITECTURE.md`) and `CHANGELOG.md`
   under `## [Unreleased]` when your change is user- or contributor-visible.
4. At least one maintainer review is required. Address feedback by pushing
   follow-up commits (don't force-push mid-review unless asked).
5. CI must be green: build, SwiftLint, SwiftFormat, the test suites, and the
   ThreadSanitizer job.

Reviews look for: correctness and safety (especially around untrusted input and
the privileged helper boundary), test coverage of boundaries and races, strict
concurrency correctness, and adherence to the module dependency rules.

## Developer Certificate of Origin (DCO)

We may ask contributors to sign off their commits under the
[Developer Certificate of Origin](https://developercertificate.org/) by adding a
`Signed-off-by` trailer:

```sh
git commit -s -m "feat(store): persist connection history"
```

This is currently **optional** but encouraged; it certifies you have the right to
submit the contribution under the project's Apache-2.0 license.

## Questions

Open an issue for anything project-related. For security reports, do **not** open
a public issue — follow [`SECURITY.md`](./SECURITY.md). General contact:
